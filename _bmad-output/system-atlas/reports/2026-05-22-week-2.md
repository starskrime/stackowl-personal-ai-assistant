## Critical Findings From Wires Pass

### Finding W-01: GatewayContext Is a Graveyard of Wired-But-Unused References

`GatewayContext` (built around line 1060 in `core.ts`) injects 12+ subsystem instances into the context object — but the actual hot path ignores most of them. The full list of GatewayContext fields that are initialized but never read downstream:

```
questManager       → QuestManager instance
capsuleManager     → CapsuleManager instance
constellationMiner → ConstellationMiner instance
timelineManager    → TimelineManager instance
socraticEngine     → SocraticEngine instance
collabSessionManager → CollabSessionManager instance
groundState        → GroundStateView instance
```

These are already wired at the dependency-injection level — adding them to the live flow requires only reading the injected references in the right handlers. Zero instantiation work needed.

### Finding W-02: A2A Is Wired Directly into GatewayCore

Agent-to-agent protocol (`A2ARegistry`) is constructor-injected into `GatewayCore` — it is not a bypass but a first-class transport peer. Parliament uses `broadcast()` to dispatch to all owl agents simultaneously. A2A replaced ACP (Agent Communication Protocol) — a 600-LoC over-engineered system that had backpressure, EventBus coupling, capability routing, and a `SessionBridgeFactory`. All of that was removed. A2A is ~262 LoC (56% reduction): pure transport, 4 files, one concern per file. `send()` never throws across the agent boundary; all paths return a typed `A2AResult<T>` discriminated union.

### Finding W-03: Telegram Menu Pre-Processing Is a Black Box

Before any message reaches `OwlGateway`, Telegram messages pass through:
1. `configMenu.handleTextInput()` — config wizard intercept
2. `TelegramRootMenu.handleTextInput()` — nav routing

If either menu "handles" the message, it never reaches the gateway. This is correct for menu navigation, but means skill injection and instinct evaluation never run for menu interactions. Menu responses are unaware of the owl's behavioral state.

### Finding W-04: Streaming Is Not Homogeneous Across Channels

| Channel | Streaming Mechanism | Gateway Sees Streaming? |
|---|---|---|
| CLI v2 | `onStreamEvent` callback → Ink re-render | YES — via `createStreamHandler()` |
| Telegram | `onStreamEvent` → `editMessageText` in-place | YES — same callback |
| Web | `ws.send({type:"stream", event})` frames | YES — WebSocket framing |
| A2A | No streaming — `A2AResult<T>` discriminated union | NO — resolved promise, 5s timeout per send() |

A2A is the only channel without streaming. Streaming is explicitly out of scope for v1; long-running tasks surface via `status:'timeout'` in the returned `A2AResult`.
