CLI Architecture Approaches

How should the CLI connect to the gateway — like Telegram does?

A

Thin Adapter + Shared Renderer Safe

NOW: CLIAdapter → owns HomeScreen + TerminalUI (duplicated rendering code) NEW: CLIAdapter ← pure transport, like TelegramAdapter ↕ GatewayMessage / GatewayResponse OwlGateway CLIAdapter owns one TerminalRenderer ├─ HomeView (was HomeScreen) ├─ SessionView (was TerminalUI) └─ shared utils (ANSI, colors, frame — no duplication)
  • Smallest change — surgical refactor of existing code
  • Adapter stays thin: normalizes keys → GatewayMessage, renders GatewayResponse
  • Eliminates all the duplicated ANSI/color/frame code
  • Same two screens, same visual design
Best if: you want clean code fast without rewriting everything
B

Component-Based UI Recommended

CLIAdapter (pure transport — same as A) ↕ OwlGateway TerminalRenderer (stateful compositor) ├─ <TopBar /> — owl badge, model, stats ├─ <LeftPanel /> — mind state, DNA, tools ├─ <RightPanel /> — conversation history ├─ <InputBox /> — prompt with cursor └─ <CmdPopup /> — /command autocomplete Each component: owns its state, renders itself, no god-class
  • Each panel is an independent, testable unit
  • Adding new panels (e.g. Parliament view) is just a new component
  • Renderer composes components — no 1000-line monolith
  • Still the same visual design, same two screens
Best if: you want the CLI to be extensible and maintainable long-term
C

Gateway-Push Architecture Large scope

CLIAdapter (pure transport — identical to Telegram) ↕ only GatewayMessage / GatewayResponse OwlGateway pushes via sendToUser() / broadcast() CLIInputHandler (separate — only captures keystrokes) TerminalRenderer (fully passive — only renders what gateway sends) Gateway drives everything. CLI never pulls. Streaming: onStreamEvent callback still works.
  • Most faithful to "connect like Telegram" in spirit
  • Renderer is completely passive — no state, just paint what gateway gives
  • Input handler is a standalone module — can be mocked/tested independently
  • Biggest architectural shift — requires rethinking session state on CLI side
Best if: you plan to support multiple concurrent CLI sessions or remote CLI
